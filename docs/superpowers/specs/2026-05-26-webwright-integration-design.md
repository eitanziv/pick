# Webwright Integration Design

**Date:** 2026-05-26
**Status:** Approved
**Origin:** StrikeKit #103 (relocated to Pick)

## Problem Statement

Modern web applications rely on JavaScript, SPAs, and complex authentication flows that cannot be tested with traditional HTTP-based tools. Pick has 28+ web tools (ffuf, sqlmap, nuclei, etc.) but none operate a real browser. This leaves gaps in testing:

- JavaScript-heavy SPAs
- OAuth2/SAML/2FA authentication flows
- DOM-based XSS
- Client-side validation bypass
- Multi-step interaction sequences
- WebSocket testing

## Solution

Integrate Microsoft's [Webwright](https://github.com/microsoft/webwright) as a sandboxed external tool within Pick. Webwright is a lightweight Python/Playwright browser agent framework (~1.5k LOC) where an LLM autonomously writes and executes browser automation scripts.

## Components

### 1. WebwrightTool (PentestTool Implementation)

Dual-mode external tool following the existing runner pattern.

**Parameters:**
- `mode`: `"explore"` | `"execute"`
- `task`: Natural language objective (explore mode)
- `script`: Python/Playwright script (execute mode)
- `start_url`: Target URL
- `max_steps`: Optional cap on agent loop iterations
- `model`: Optional model tier override

**Explore mode:** Autonomous. Webwright's LLM loop drives the browser, decides what to test, produces scripts + findings. Used for reconnaissance, auth flow mapping, vulnerability discovery.

**Execute mode:** Scripted. Replays a Playwright script (e.g., one previously generated by explore mode) for validation/evidence capture. Used for exploit confirmation and regression testing.

### 2. LLM Routing via Strike48 Conversations

Webwright's LLM calls are mapped to messages in a Strike48 conversation rather than direct API calls.

**Flow:**
1. `WebwrightTool` creates/reuses a conversation with a "Browser Automation" persona
2. Webwright makes OpenAI-format requests to a local proxy (`http://127.0.0.1:{port}/v1/chat/completions`)
3. Pick's proxy translates each request into a conversation message sent via the connector SDK (`invoke_request`)
4. Strike48's ConversationServer processes via GenAssistant (system prompt, model, tools)
5. Response flows back through the stream, proxy formats as OpenAI response

**Benefits:**
- All LLM interactions are auditable in Strike48's conversation history
- Persona system prompt is configurable server-side without redeploying Pick
- Token usage tracked through normal Strike48 metering
- Conversation context builds naturally (agent "remembers" prior exploration)
- Operators can watch the browser agent's reasoning in real-time in Strike48 UI

**Persona configuration:**
```
name: "Browser Recon Agent"
model: :smart
system_message: "You are a web security testing agent controlling a browser via
  Playwright. Generate Python scripts to explore and test web applications. Focus
  on discovering authentication flows, injection points, and client-side
  vulnerabilities..."
tools: { system_tools: [], client_tools: ["webwright_execute_script"] }
```

**Webwright config:**
```yaml
model:
  provider: openai
  base_url: http://127.0.0.1:{port}/v1
  api_key: "not-needed-but-required-field"
  model: "strike48-default"
```

### 3. Evidence Collection

Everything Webwright produces becomes an `EvidenceNode` with provenance.

| Artifact | Evidence Type | Purpose |
|----------|--------------|---------|
| `*.py` scripts | Exploit / Technique | Replayable proof, shareable |
| `screenshot_*.png` | Observation | Visual state at key moments |
| `network_log.json` | Observation | All HTTP/WS requests+responses |
| `console_output.log` | Observation | Browser JS console |
| `dom_snapshot_*.html` | Observation | DOM state when finding triggered |
| `findings.json` | Finding | Structured vulnerability data |
| Agent reasoning log | Provenance | LLM decision trace |

**Evidence chain mapping:**
```
Observation (screenshot, network log, DOM snapshot)
    -> Hypothesis ("form at /login may be vulnerable to reflected XSS")
    -> Exploit (generated Python script that demonstrates it)
    -> Finding (confirmed XSS, severity, remediation)
```

**Provenance tags:** task ID, timestamp, Webwright version, model used, agent loop iteration count, conversation ID.

**Storage:** `evidence/webwright/{task_id}/` in Pick's workspace, ingested via `push_evidence()`.

### 4. Sandbox Execution & Lifecycle

**Execution flow:**
1. Create workspace dir: `/tmp/webwright/{task_id}/`
2. Write config.yaml (base_url, model, task)
3. Start local proxy (if not already running)
4. Create/reuse conversation via connector SDK
5. Spawn in sandbox: `proot/bwrap python -m webwright.run.cli -c /workspace/config.yaml -t "{task}" --start-url {url} --workspace /workspace/`
6. Monitor workspace for artifacts (poll or inotify)
7. Emit terminal lines to UI as Webwright progresses
8. On exit: sweep workspace, map to EvidenceNodes, push to evidence buffer, return ToolResult

**Lifecycle:**
- **Chromium reuse:** For multi-task engagements, keep sandbox alive between tasks (warm browser). Kill on engagement end or idle timeout.
- **Cleanup:** Workspace dirs cleaned after evidence is ingested. Sandbox torn down on connector shutdown.
- **Resource limits:** bwrap/proot caps memory/CPU. Defaults: 512MB-1GB for Chromium.
- **Failure modes:** Crash/timeout -> partial artifacts still collected. Conversation history on Strike48 preserves reasoning up to failure.
- **Timeout:** Configurable per-task (max_steps or wall-clock).

### 5. Specialist Agent Integration

Pick's Web App Specialist dispatches Webwright based on task type.

**Dispatch patterns:**

- **Early engagement (explore):** "Map the application, identify auth flows, enumerate endpoints, fingerprint technologies"
- **Mid engagement (targeted explore):** "Test form inputs on /login for XSS", "Attempt OAuth flow bypass"
- **Validation (execute):** Replay generated script, capture fresh evidence, confirm findings

**Relationship to existing tools:**

Webwright does NOT replace ffuf, nuclei, sqlmap. Those are fast and well-understood for known patterns. Webwright handles what they cannot:
- JavaScript-heavy SPAs needing a real browser
- Complex auth flows (OAuth, SAML, 2FA)
- DOM-based vulnerabilities
- Multi-step interaction sequences
- Anything requiring "a human would click through this"

The specialist decides: traditional tools for known patterns, Webwright for dynamic/complex scenarios.

## Stretch Goal: Sidecar Mode for Real-Time Updates

After the initial external tool integration ships, evolve to a persistent sidecar for richer feedback.

**Motivation:**
- Explore tasks run for minutes (76 steps average on benchmarks)
- Users want live progress, not silence then artifact dump
- Intermediate screenshots and findings have value before completion

**Protocol (JSON-over-stdin/stdout):**

```
Pick -> Sidecar:
  { "type": "start_task", "mode": "explore", "task": "...", "url": "..." }
  { "type": "execute_script", "script": "..." }
  { "type": "cancel" }

Sidecar -> Pick:
  { "type": "step", "n": 3, "action": "navigating to /login", "screenshot": "base64..." }
  { "type": "finding", "severity": "high", "detail": "..." }
  { "type": "script_generated", "path": "/workspace/exploit_xss.py" }
  { "type": "network_event", "request": {...}, "response": {...} }
  { "type": "complete", "summary": "...", "artifacts": [...] }
```

**Enables:**
- Live screenshots in UI (operators see what browser sees, step by step)
- Early findings surfaced before task completion
- Interactive steering (future: operator sends "focus on that form" mid-task)
- Warm browser (no Chromium cold-start between tasks)
- Network event streaming for real-time traffic inspection

**Implementation sequence:**
1. Ship subprocess-per-task approach (Approach A)
2. Wrap Webwright CLI with thin Python stdin/stdout server (~100 lines)
3. Update `WebwrightTool` to prefer sidecar if running, fall back to subprocess
4. Future: interactive steering from operators

## Implementation Priority

1. `WebwrightTool` struct + schema + sandbox spawning (execute mode first)
2. Local LLM proxy route (OpenAI format -> conversation message -> response)
3. Explore mode (full autonomous agent loop)
4. Evidence collection pipeline (workspace sweep -> EvidenceNodes)
5. Specialist agent dispatch integration
6. Stretch: sidecar mode

## Dependencies

- Python 3.10+ in sandbox (already available in BlackArch rootfs)
- Playwright + Chromium (pip install in sandbox)
- Webwright (pip install or vendor)
- Axum route for local LLM proxy (Pick already runs axum)
- Connector SDK invoke_request for conversation messaging (already used)

## Non-Goals

- Replacing existing HTTP-based web tools
- Running Webwright outside the sandbox
- Direct LLM API access from device (all calls route through Strike48)
- Full Webwright fork/rewrite in Rust
